信息安全方案

1. 代码规范：静态检查通过 ruff 强制执行（pyproject.toml 配置 E/F/I/N/W/UP 规则集），CI 流水线（.github/workflows/quality-gate.yml）在合并前自动执行 ruff check 与 pytest，未通过则阻断合并；涉及权限/审计等安全相关变更需先产出 RFC 设计文档评审（如 docs/rfcs/0001-rbac-phase1.md）再实施。

2. 权限安全：严格遵循"最小权限原则"，基于 RBAC 模型实现细粒度访问控制——内置 9 种角色，划分 platform / org / workspace / instance 四级 scope（app/models/rbac/role.py），通过 require_perms 装饰器在 API 层强制校验（app/core/rbac/decorators.py）；人类用户走 JWT 鉴权（get_current_user），AI 员工走 proxy_token（get_current_user_or_agent），两类身份权限边界隔离。所有权限决策与操作行为分别记录于 permission_audit_log 与 operation_audit_log（均为 append-only 不可篡改），完善审计能力。

3. 安全漏洞检查：登录接口内置基于滑动窗口的速率限制（app/core/auth_rate_limit.py，按 IP + 账号双维度、5 次/60 秒），防止暴力破解；服务启动时强制校验 JWT_SECRET / ENCRYPTION_KEY 等关键密钥不得为默认值，生产环境检测到默认值将拒绝启动。当前限流为进程内内存实现，多副本部署下为近似限流，尚未引入 Redis 做跨实例共享；常态化自动化漏洞扫描（SAST/DAST）与专业渗透测试机制尚未建立，需纳入后续规划补齐。

4. 数据安全：敏感数据（如 K8s KubeConfig）采用 AES-256-GCM 算法加密存储（encrypt_kubeconfig / decrypt_kubeconfig，app/core/security.py），密钥经 ENCRYPTION_KEY 环境变量管理，与代码库和数据库分离；接口层通过 CORSMiddleware 严格限制跨域来源白名单（默认仅本地开发域名，生产环境需显式配置）。当前仓库内 K8s Ingress 未配置 TLS（deploy/k8s/ingress.yaml 无 tls 字段），传输层加密依赖外部网关/证书管理落地，建议在部署文档中明确要求生产环境必须启用 HTTPS。

5. 数据合规：全局遵循软删除规范，BaseModel 统一提供 deleted_at 字段与 soft_delete() 方法，唯一约束使用 Partial Unique Index，避免物理删除导致数据不可追溯；代码与测试数据中的个人信息（如邮箱）统一使用 @example.com 占位，禁止出现真实用户信息（项目 CLAUDE.md 明确约定）。数据分级分类制度与定期合规自查机制目前尚未形成文档化流程，建议后续补充。
如实说明：以下为当前尚未落地、建议后续规划补齐的项 自动化 SAST/DAST 代码扫描与专业渗透测试机制未建立；速率限制为单进程内存实现，未做跨副本共享；仓库内未见 Ingress TLS 配置，生产环境 HTTPS 依赖外部证书管理；数据分级分类与定期合规自查暂无文档化流程。以上均基于对当前代码库的实际调研得出，未做推测性美化。
