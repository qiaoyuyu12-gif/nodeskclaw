# NoDeskClaw 手动启动指南（Windows / Linux / WSL）

> 适用场景：Windows 本机开发，通过三个终端窗口手动启动
> backend / llm-proxy / portal 三个服务。
>
> **为什么不用 `./dev.sh`**：MSYS/Git Bash 下 bash 后台管道（`cmd | sed &`）的 PID
> 跟踪和真实 Linux 不一致，`dev.sh` 的进程管理会误判子进程已死而立即退出。
> 手动起三个窗口最稳，且每个服务的报错都直接显示在自己窗口，调试最快。
>
> Linux/macOS 可直接用 `./dev.sh ce` 或 `./dev.sh ee`。
>
> **Windows 终端选择**：
> - **Backend**：必须用 **PowerShell** 或 cmd，**不能用 Git Bash**
>   （Git Bash 下 uvicorn 的 `--reload` 子进程无法正常接管 socket，进程表现为
>   "启动成功"但所有 HTTP 请求超时；且 Ctrl+C 后 reloader 主进程残留，导致端口
>   无法重新绑定）
> - **LLM Proxy / Portal**：Git Bash 或 PowerShell 均可

---

## Linux / WSL 启动（前置准备与注意事项）

> 适用场景：在 Linux，或 Windows 上的 **WSL2** 里启动。WSL2 是真正的 Linux 内核，
> 上文所有 Windows/Git Bash 专属的坑（`SKIP_AUTO_MIGRATE`、backend 必须用 PowerShell、
> `--reload` socket 损坏、`dev.sh` PID 误判）**全部不适用**。直接用 `./dev.sh ce`（或 `ee`）即可。
>
> 下面是在一台**全新 WSL2（Ubuntu）** 上从零跑起来的完整前置准备，每条都对应实际踩过的坑。

### L0. 全局注意事项

- **项目放 WSL 原生文件系统，不要放 `/mnt/<盘符>`**：`/mnt/f` 等是 9p 跨盘挂载，
  `uv sync` / `npm install` 极慢，且 Vite 热更新和 uvicorn `--reload` 的文件监听经常失效。
  正确做法：`cp -r /mnt/f/nodeskclaw ~/nodeskclaw`，之后都在 `~/nodeskclaw` 里操作。
- **命令一次只粘一行**：多行一起粘贴时，第一行之外的命令常被终端吞掉（建库/装包看似成功实则没执行），
  本指南里反复出现的"database does not exist""依赖没装上"多半源于此。

### L1. 安装工具链

```bash
# uv（Python 包管理器）
curl -LsSf https://astral.sh/uv/install.sh | sh
source $HOME/.local/bin/env        # 确保 uv 进入当前 shell 的 PATH
uv --version

# Node.js ≥ 18 —— 用 apt，不要用 nvm
sudo apt update
sudo apt install -y nodejs npm
hash -r
which node                          # 必须是 /usr/bin/node（不是 /mnt/c/...）
node --version
```

> **不要用 nvm**：`nvm` 安装脚本会 `git clone` GitHub，国内网络常报
> `GnuTLS recv error (-110): The TLS connection was non-properly terminated` 而失败。
> apt 装的是原生 Linux Node，最稳。
>
> **警惕 Windows Node 泄漏进 WSL 的 PATH**：若 `which node`/`which npm` 指向 `/mnt/c/nvm4w/...`，
> 说明用的是 Windows 的 node.exe，不可用。装完 apt 版后 `which node` 必须是 `/usr/bin/node`。

### L2. 安装并初始化 PostgreSQL（无 Docker Desktop 时用原生 PG）

WSL 默认没有 Docker Desktop，`./dev.sh --docker-pg` 走不通。直接用 apt 装原生 PostgreSQL：

```bash
sudo apt install -y postgresql
sudo systemctl enable --now postgresql       # systemd 环境；或 sudo service postgresql start
```

逐行执行建用户、建库（密码、库名必须与 `.env` 的 `DATABASE_URL` 完全一致）：

```bash
sudo -u postgres psql -c "CREATE USER nodeskclaw WITH PASSWORD 'nodeskclaw123';"
sudo -u postgres psql -c "ALTER USER nodeskclaw WITH PASSWORD 'nodeskclaw123';"
sudo -u postgres psql -c "CREATE DATABASE nodeskclaw_rbac OWNER nodeskclaw;"
sudo -u postgres psql -c "GRANT ALL PRIVILEGES ON DATABASE nodeskclaw_rbac TO nodeskclaw;"
```

验证 TCP 连接能通（这步成功才算真的好了）：

```bash
PGPASSWORD=nodeskclaw123 psql -h localhost -U nodeskclaw -d nodeskclaw_rbac -c "select 1;"
```

> 若报 `password authentication failed` → 用户没建成功，重跑 `CREATE/ALTER USER`。
> 若报 `database ... does not exist` → 库没建成功（多半多行粘贴被吞），单独重跑 `CREATE DATABASE`。

### L3. ⚠️ 配置 `.env`：务必去掉 CRLF（最隐蔽的坑）

从 Windows 复制/克隆来的 `.env`（或 `cp .env.example`）常是 **CRLF 换行**。
`dev.sh` 用 `grep | cut` 从 `.env` 提取 `DATABASE_URL` 并 `export`，会把行尾的 `\r` 一起带上，
导致数据库名变成 `nodeskclaw_rbac\r`，连接报 **database "nodeskclaw_rbac" does not exist**，
且 `\r` 会让终端光标回到行首覆盖文字，错误信息显示成乱码（如 `" does not existeskclaw_rbac...`）。

> 诡异表象：直接 `uv run alembic upgrade head` 能连上（pydantic 读 `.env` 会自动去 `\r`），
> 但经 `dev.sh` 启动的 backend 内部迁移失败 —— "外层能连、内部不能"就是这个原因。

```bash
cp nodeskclaw-backend/.env.example nodeskclaw-backend/.env
sed -i 's/\r$//' nodeskclaw-backend/.env        # 去掉所有行尾 CR
grep -c $'\r' nodeskclaw-backend/.env           # 应输出 0
```

`.env` 关键项（Linux 上无需 `SKIP_AUTO_MIGRATE`、无需 `NODESKCLAW_DATA_DIR`）：

```env
DATABASE_URL=postgresql+asyncpg://nodeskclaw:nodeskclaw123@localhost:5432/nodeskclaw_rbac
NODESKCLAW_EDITION=ce
LLM_PROXY_URL=http://localhost:4511
LLM_PROXY_INTERNAL_URL=http://localhost:4511
JWT_SECRET=change-me-in-production
ENCRYPTION_KEY=change-me-32-bytes-base64-key__=
```

### L4. 启动

```bash
cd ~/nodeskclaw
./dev.sh ce        # 或 ./dev.sh ee（需 ee/ 目录存在；组织审核等 multi_org 功能仅 EE 可见）
```

成功标志：彩色 banner + backend 打印超管账号/初始密码 + 无 `ECONNREFUSED 127.0.0.1:4510`。
浏览器开 http://localhost:4517 登录。

### L5. 安装 Docker（创建「Docker 类型 AI 员工」实例所需）

平台创建 Docker 运行环境时会调 `docker compose version`，WSL 默认没装 docker 会报
**「Docker CLI 未安装」**。在 WSL 里装原生 Docker Engine：

```bash
sudo apt install -y docker.io docker-compose-v2
sudo systemctl enable --now docker          # WSL 需 systemd（/etc/wsl.conf 含 [boot] systemd=true）
sudo usermod -aG docker $USER               # 免 sudo 访问 docker.sock
```

让 docker 组生效（`newgrp` 在精简镜像里可能没有）——**在 Windows 的 PowerShell/CMD** 里重启 WSL：

```powershell
wsl --shutdown
```

> `wsl` 是 Windows 命令，**不能在 WSL 内部跑**。重启后重新打开 WSL 终端，docker 组即生效。
> PostgreSQL / docker 守护进程（已 `systemctl enable`）会随 WSL 重启自动拉起。

验证（逐行）：

```bash
docker compose version      # 输出版本 = 后端检查能过
docker ps                   # 不报 permission denied = docker 组已生效
```

> 若 `docker ps` 仍报 `permission denied while trying to connect to ... docker.sock`，
> 说明 WSL 没真正重启或组没生效，重新 `wsl --shutdown` 后再开新窗口。

### L6. 镜像拉取（国内网络加速）

创建 AI 员工实例时后端会从 Docker Hub（`registry-1.docker.io`）拉镜像。国内网络常超时
（现象：实例长期 pending、`i/o timeout`、`pull ... TLS handshake timeout`）。配镜像加速：

```bash
sudo tee /etc/docker/daemon.json > /dev/null <<'EOF'
{
  "registry-mirrors": ["https://docker.m.daocloud.io", "https://dockerproxy.com"]
}
EOF
sudo systemctl restart docker
docker info | grep -A3 "Registry Mirrors"
```

> 镜像加速地址时效性强，若上述失效请换用当时可用的公共镜像源。

## 2. 配置 `nodeskclaw-backend/.env`（一次性）

```env
# 数据库（密码 nodeskclaw123，库名 nodeskclaw_rbac）
DATABASE_URL=postgresql+asyncpg://nodeskclaw:nodeskclaw123@localhost:5432/nodeskclaw_rbac

# 版本选择：ce（社区版）或 ee（企业版）
NODESKCLAW_EDITION=ce

# LLM Proxy 地址（手动启动必须显式设置，dev.sh 会自动注入）
LLM_PROXY_URL=http://localhost:4511
LLM_PROXY_INTERNAL_URL=http://localhost:4511

# 超管初始账号（首次启动后可改密码，改回 false 则不再重置）
INIT_ADMIN_ACCOUNT=admin
RESET_ADMIN_PASSWORD=false

# EE 模式额外账号（仅 ee 时有效）
INIT_EE_ADMIN_ACCOUNT=deskclaw-admin
RESET_EE_ADMIN_PASSWORD=false

# JWT / 加密密钥（生产务必替换）
JWT_SECRET=change-me-in-production
ENCRYPTION_KEY=change-me-32-bytes-base64-key__=

# CORS：本机及局域网（私网网段）默认即可访问，无需逐个 IP 手填
# 后端 CORS_ORIGIN_REGEX 已默认放行 10.* / 192.168.* / 172.16-31.* 的任意端口来源
CORS_ORIGINS=["http://localhost:4517","http://localhost:4518"]
# 若访问来源不在私网网段（如公网/特殊网段），再按需追加：
# CORS_ORIGINS=["http://localhost:4517","http://localhost:4518","http://<访问来源>:4517"]

# Windows 首次启动：先手动跑迁移（见第 3 节），再加这行跳过启动时自动迁移
# （Windows 上 Alembic 在 lifespan 内 to_thread 运行会卡死，导致所有 HTTP 请求超时）
SKIP_AUTO_MIGRATE=1
```

## 4. 启动三个服务（三个终端窗口）

> **启动顺序**：llm-proxy → backend → portal。

### 窗口 ① — LLM Proxy（端口 4511）

Git Bash 或 PowerShell 均可：

```bash
cd /f/nodeskclaw/nodeskclaw-llm-proxy
# DATABASE_URL 必须与 backend 保持一致（.env 里已配置时可省略 export）
export DATABASE_URL="postgresql+asyncpg://nodeskclaw:nodeskclaw123@localhost:5432/nodeskclaw_rbac"
uv run uvicorn app.main:app --host 0.0.0.0 --port 4511
```

看到 `Uvicorn running on http://0.0.0.0:4511` 即就绪。

> **注意**：若 LLM Proxy 的 DATABASE_URL 指向旧库（`nodeskclaw`），
> OpenClaw 发来的 LLM 请求会返回 401 Unauthorized。
> 确保两个服务连接同一个数据库 `nodeskclaw_rbac`。

### 窗口 ② — Backend（端口 4510）⚠️ 必须用 PowerShell

```powershell
cd F:\nodeskclaw\nodeskclaw-backend
# SKIP_AUTO_MIGRATE=1：Windows 上必须加，否则 Alembic 在 lifespan 内卡死导致请求超时
# 不加 --reload：Git Bash 下 --reload 子进程 socket 传递损坏；PowerShell 下可加但非必须
$env:SKIP_AUTO_MIGRATE="1"
uv run uvicorn app.main:app --host 0.0.0.0 --port 4510
```

启动成功标志：

```
INFO  FeatureGate: edition=ce
INFO  Application startup complete.
INFO  Uvicorn running on http://0.0.0.0:4510
```

**留意日志里打印的超管账号/初始密码**，登录要用。

### 窗口 ③ — Portal（端口 4517）

Git Bash 或 PowerShell 均可：

```bash
cd /f/nodeskclaw/nodeskclaw-portal
npm install   # 第一次或 package.json 有变更时执行
npm run dev
```

看到 `Local:   http://localhost:4517/` 即就绪。局域网访问地址见 `Network:` 那行。

## 7. 停止服务

每个窗口 `Ctrl+C` 即可。如有残留进程，**在 PowerShell 里**：

```powershell
# 查端口占用（以 4510 为例）
netstat -ano | findstr :4510

# 用上一步输出的 PID 杀掉（可指定多个）
taskkill /PID <PID> /F

# 或一次性杀掉所有 Python 进程（慎用，会连带 llm-proxy 等）
Get-Process python | Stop-Process -Force
```

> **Git Bash 下 Ctrl+C 的陷阱**：Git Bash 里运行的 uvicorn `--reload`，Ctrl+C 只杀了
> worker 子进程，**reloader 主进程仍在运行并持有端口**。重启 backend 前必须用上面的
> PowerShell 命令确认端口已释放，否则新进程会启动成功但无法绑定端口。

PostgreSQL：

```bash
docker stop nodeskclaw-postgres   # 停止
docker start nodeskclaw-postgres  # 恢复
```

## 8. 常见问题

### Q1: `[dev] ERROR: 未找到 uv`

Git Bash 没继承 PATH。按本文「前置条件」节把 PATH 写入 `~/.bashrc` 即可。

### Q2: `LLM_PROXY_URL 未配置，服务无法启动`

`.env` 里缺 `LLM_PROXY_URL=http://localhost:4511`。按本文「2. 配置 .env」补上。

### Q3: 数据库连接失败，URL 里出现 `<host>`

`.env` 里 `DATABASE_URL` 还是占位符模板，按本文「2. 配置 .env」改成实际地址。

### Q4: OpenClaw 发给 LLM Proxy 的请求返回 401

LLM Proxy 连接的数据库与 Backend 不一致，查不到实例 token。
确保两个服务的 `DATABASE_URL` 完全相同：
```
postgresql+asyncpg://nodeskclaw:nodeskclaw123@localhost:5432/nodeskclaw_rbac
```

### Q5: PowerShell 调 `& bash dev.sh` 后服务很快全部退出

PowerShell 通过 `&` 启动 bash 时，后台进程没有控制终端，会被立即回收。
**必须用真正的 Git Bash 终端**，从开始菜单或文件资源管理器右键「Open Git Bash here」打开。

### Q6: `./dev.sh` 跑到「启动服务...」就退出，看不到 banner

MSYS bash 的后台管道 PID 跟踪 bug。**按本文三窗口手动启**，不要用 `dev.sh`。
（Linux/macOS 下 `./dev.sh ce` 是正常的，不受此影响。）

### Q7: 端口已被占用

在 PowerShell 里：

```powershell
netstat -ano | findstr ":4510"   # 查 PID
taskkill /PID <PID> /F           # 杀掉
```

### Q8: 依赖未安装

```bash
cd nodeskclaw-backend     && uv sync
cd nodeskclaw-llm-proxy   && uv sync
cd nodeskclaw-portal      && npm install
```

### Q9: 复制粘贴命令报 `bash: $'\302\226': command not found`

复制时带了不可见 Unicode 控制字符。在 Git Bash 里**手动键入**命令，或先 `clear` 再粘贴。

### Q10: 部署 Docker 类型 AI 员工失败（Windows 专属）

Windows 下必须在后端 `.env` 中显式设置数据目录：

```env
NODESKCLAW_DATA_DIR=C:\Users\gg102\.nodeskclaw\docker-instances
```

### Q11: Backend 启动成功但登录/所有接口超时（Windows 专属）

**现象**：日志打出 `Application startup complete`，端口 4510 有监听，但所有 HTTP 请求挂起无响应。

**原因 A — Alembic 自动迁移卡死**：backend 启动时会在 `lifespan` 内通过 `asyncio.to_thread` 跑 Alembic，Windows 上此调用偶发卡死，导致 lifespan 永不 yield，uvicorn 接受 TCP 连接但不处理 HTTP。

> 解决：`.env` 加 `SKIP_AUTO_MIGRATE=1`，并在启动前手动跑 `uv run alembic upgrade head`（见第 3 节）。

**原因 B — Git Bash 下 `--reload` 子进程 socket 损坏**：Git Bash（MinGW）下 uvicorn 用子进程传递 socket 给 worker，Windows 原生 socket handle 在 MSYS 层损坏，worker 完成 lifespan 但无法服务 HTTP。

> 解决：backend 改用 **PowerShell**（或 cmd）启动，去掉 `--reload`。

### Q12: `[Errno 10048]` 端口无法绑定，但 netstat 显示端口空闲或只有 TIME_WAIT

Git Bash 里的 uvicorn `--reload` Ctrl+C 后，**reloader 主进程没被杀**，仍持有 4510 端口。`TIME_WAIT` 状态（PID=0）是正常的 TCP 回收残留，不阻止绑定；真正阻止绑定的是那个残留主进程。

```powershell
# 找出所有 Python 进程及启动时间
Get-Process python | Select-Object Id, StartTime

# 杀掉所有（或根据 StartTime 只杀旧的）
Get-Process python | Stop-Process -Force

# 确认端口已释放
netstat -ano | findstr :4510
```

### Q13: 局域网其他设备访问 Portal

三个服务（backend 4510 / llm-proxy 4511 / portal 4517）默认就监听 `0.0.0.0`
（backend、llm-proxy 用 `uvicorn --host 0.0.0.0`，portal 的 Vite 配了 `server.host: true`），
**不是只能本机访问**。其他设备访问通常只差一步：放行防火墙。

1. **放行 Windows 防火墙**（管理员 PowerShell，一次性）——这是 Windows 上最常见的真实阻断点：
   ```powershell
   New-NetFirewallRule -DisplayName "NoDeskClaw Portal 4517" -Direction Inbound -Action Allow -Protocol TCP -LocalPort 4517
   # 若其他设备需直连后端（如访问 /docs），再放行 4510：
   New-NetFirewallRule -DisplayName "NoDeskClaw Backend 4510" -Direction Inbound -Action Allow -Protocol TCP -LocalPort 4510
   ```
2. 从局域网设备访问 `http://<LAN-IP>:4517`（LAN IP 通过 `ipconfig` 查以太网适配器的 IPv4 地址，
   或直接看 portal 启动日志里 Vite 打印的 `Network:` 那行）。
3. **关于 CORS**：常规链路里浏览器只访问 portal(4517)，`/api` 由 Vite 代理转发到后端属同源请求，
   不触发 CORS。后端 `CORS_ORIGIN_REGEX` 还默认放行了私网网段（`10.*` / `192.168.*` / `172.16-31.*`）
   的任意端口来源，因此局域网设备即使直连后端 4510 也不会报跨域，**无需再逐个 IP 手填 `CORS_ORIGINS`**。
   仅当需要对外网严格收紧时，才在 `.env` 把 `CORS_ORIGIN_REGEX` 置空关闭该正则。

## 9. EE 模式启动（企业版）

1. 确认 `ee/` 目录存在（私有子仓库已 clone）
2. 改 `.env`：`NODESKCLAW_EDITION=ee`
3. 重启 backend 窗口
4. Portal 不需重启，`/api/v1/system/info` 返回 `ee` 后前端自动解锁 EE 功能
