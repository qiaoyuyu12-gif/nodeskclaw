## 1. 系统概述

MOM Cloud 是一套基于微服务架构的制造运营管理平台，采用**门户 + 子系统**的多应用架构。平台包含 6 个子系统：

| 子系统 | 应用编码 | 入口路径 | 说明 |
|--------|----------|----------|------|
| APS 高级计划排程 | `APP_APS` | `/aps/home` | 排产管理、交付策划、产能规划 |
| MES 制造执行 | `APP_MES` | `/mes/home` | 生产工单、工序执行、质量检验 |
| WMS 仓储管理 | `APP_WMS` | `/wms/home` | 仓储管理、物料调度 |
| QMS 质量管理 | `APP_QMS` | `/qms/home` | 质量检验、缺陷管理 |
| EAM 设备管理 | `APP_EAM` | `/eam/home` | 设备资产管理、维保计划 |
| TMS 追溯管理 | `APP_TMS` | `/tms/home` | 生产追溯、批次跟踪 |

权限控制贯穿认证、路由、菜单、按钮、数据五个层级，实现从入口到数据的完整管控。

## 2. 权限模型设计

### 2.1 RBAC 权限模型

系统采用 **RBAC（基于角色的访问控制）+ 应用层** 扩展模型：

```
用户 (sys_user)
  │
  ├── N:M ──→ 角色 (sys_user_role) ──→ 角色 (sys_role)
  │                                       │
  │                                       ├── N:M ──→ 菜单/权限 (sys_role_menu) ──→ 菜单 (sys_menu)
  │                                       │
  │                                       └── N:M ──→ 应用 (sys_role_app) ──→ 应用 (sys_app)
  │
  └── N:1 ──→ 部门 (sys_dept) [树形层级]
```

### 2.2 五层权限控制体系

| 层级 | 控制对象 | 实现方式 | 作用域 |
|------|----------|----------|--------|
| **L1 认证层** | 用户身份 | JWT Token + Redis 会话 | 网关全局 |
| **L2 应用层** | 子系统入口 | `sys_role_app` 关联 + 前端路由守卫 | 门户→子系统 |
| **L3 菜单层** | 侧边栏菜单 | `sys_role_menu` 关联 + 前端路由 meta.roles | 子系统内部 |
| **L4 按钮层** | 页面操作按钮 | `sys_menu.perms` (type=F) + `useAuth()` | 页面内部 |
| **L5 数据层** | 数据行过滤 | `tenant_id` 自动注入 + `data_scope` | SQL 级别 |

### 2.3 权限解析链路

```
用户登录
  → 查询 sys_user_role → 获取角色列表
  → 查询 sys_role_menu → 获取菜单权限 (perms 字段)
  → 查询 sys_role_app  → 获取可访问应用列表
  → 组装 LoginUser {userId, roles[], permissions[], apps[]}
  → 存入 Redis (login_token:{userId})
  → 签发 JWT Token 返回前端
```

### 4.2 JWT Token 结构

```
Header:  { alg: "HS256", typ: "JWT" }
Payload: {
  sub: "admin",              // 用户名
  user_id: 1,                // 用户ID
  token_id: "uuid-xxx",      // 令牌唯一标识
  tenant_id: 1,              // 租户ID
  exp: 1713100800,           // 过期时间 (12小时)
  iat: 1713057600            // 签发时间
}
Signature: HMACSHA256(header + payload, secret)
```

- **签名密钥**: 环境变量 `MOM_JWT_SECRET` 配置
- **有效期**: 720 分钟 (12小时)，通过 `mom.jwt.expiration` 可配
- **签名算法**: HMAC-SHA256

### 4.3 Redis 会话存储

```
Key:   login_token:{userId}
Value: LoginUser {
         userId, username, nickname,
         tenantId, deptId,
         roles: ["admin"],
         permissions: ["system:user:list", "system:user:add", ...]
       }
TTL:   720 minutes (与 JWT 同步)
```

### 4.4 Token 刷新机制

```
前端请求 → 后端返回过期错误码 (VITE_SERVICE_EXPIRED_TOKEN_CODES)
  → 前端调用 POST /auth/refresh {refreshToken}
  → 后端验证 refreshToken → 签发新 JWT → 更新 Redis TTL
  → 前端存储新 token → 重发原始请求
```

## 5. 鉴权机制

### 5.1 网关层鉴权 (L1)

**组件**: `AuthFilter.java` (Gateway GlobalFilter, order=-100)

**白名单路径** (无需认证):
- `/auth/login`, `/auth/logout`, `/auth/captcha`, `/auth/register`
- `/auth/info`, `/auth/refresh`
- `/v3/api-docs/**`, `/swagger-ui/**`, `/doc.html/**`

**处理流程**:
1. 检查路径是否在白名单
2. 提取 `Authorization: Bearer {token}` 头
3. 解析并验证 JWT 签名
4. **安全防护**: 清除请求中的 `X-User-Id`、`X-User-Name`、`X-Tenant-Id` 头 (防伪造)
5. 从 JWT 中提取信息，注入内部请求头
6. Token 无效返回 **401**

### 5.2 微服务层鉴权 (L4)

**组件**: `TokenFilter.java` (每个微服务的 Spring Security 过滤器)

**处理流程**:
1. 读取网关注入的 `X-User-Id` 请求头
2. 从 Redis 加载 `LoginUser` 对象
3. 存入 `ThreadLocal` (`SecurityUtils.setLoginUser`)
4. 设置 Spring Security 的 `SecurityContextHolder`

### 5.3 接口权限注解 (L4)

**组件**: `@RequiresPermission` 注解 + `PermissionAspect` 切面

```java
// 注解定义
@Target({ElementType.METHOD, ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
public @interface RequiresPermission {
    String value();           // 权限标识 (如 "system:user:add")
    boolean all() default false;  // true=需全部匹配, false=任一匹配
}
```

**切面检查逻辑**:
```
请求进入 → @RequiresPermission("system:user:add")
  → 检查是否登录 (LoginUser != null), 否则抛 401
  → 检查是否超级管理员 (isAdmin()), 是则放行
  → 检查用户 permissions 是否包含 "system:user:add"
  → 不包含则抛 403 "没有权限: system:user:add"
```

### 5.4 权限标识命名规范

```
{模块}:{资源}:{操作}

示例:
  system:user:list      用户列表查询
  system:user:add       用户新增
  system:user:edit      用户编辑
  system:user:remove    用户删除
  system:role:list      角色列表查询
  system:menu:list      菜单列表查询
  system:dict:list      字典列表查询
```

## 6. 前端权限控制

### 6.1 路由权限 (L2+L3)

前端采用 **静态路由模式** (`VITE_AUTH_ROUTE_MODE=static`)，路由权限通过 `meta.roles` 声明。

**路由过滤链**:
```
全量静态路由定义 (routes.ts)
  → filterAuthRoutesByRoles()     按用户角色过滤
  → filterAuthRoutesByAppCode()   按当前子系统过滤
  → getGlobalMenusByAuthRoutes()  生成侧边栏菜单
  → addRoutesToVueRouter()        注册到 Vue Router
```

**路由 Meta 权限字段**:

| 字段 | 类型 | 说明 |
|------|------|------|
| `roles` | `string[]` | 允许访问的角色列表 (任一匹配即可), 为空表示不限制 |
| `appCode` | `string` | 所属子系统编码, 为空表示所有子系统共享 |
| `constant` | `boolean` | 公开路由, 无需登录 (如 login, 404) |
| `hideInMenu` | `boolean` | 是否在侧边栏隐藏 |

**路由配置示例**:
```typescript
{
  name: 'basic-data',
  path: '/basic-data',
  meta: {
    title: '基础数据',
    roles: ['admin', 'aps_admin', 'aps_user'],  // 角色限制
    appCode: 'APP_APS',                          // 仅APS子系统显示
    order: 2
  }
}
```

### 6.2 路由守卫体系

| 守卫 | 类型 | 说明 |
|------|------|------|
| `createRouteGuard` | beforeEach | 核心守卫: 登录检查→路由初始化→角色校验→应用权限 |
| `createAppContextGuard` | afterEach | 自动检测当前子系统, 设置 `currentAppCode` |
| `createProgressGuard` | beforeEach/afterEach | NProgress 进度条 |
| `createDocumentTitleGuard` | afterEach | 更新页面标题 |

**核心路由守卫逻辑**:
```
beforeEach(to, from, next):
  1. 初始化常量路由 (首次)
  2. 用户未登录 → 跳转 /login
  3. 初始化认证路由 (首次)
  4. 检查路由 meta.roles → 无权限跳转 403
  5. 检查子系统权限 (appRouteMap) → 无权限跳转 /portal
  6. 正常放行
```

### 6.3 按钮权限 (L4)

通过 `useAuth()` 组合式函数实现:

```typescript
// hooks/business/auth.ts
export function useAuth() {
  const authStore = useAuthStore();

  function hasAuth(codes: string | string[]) {
    if (!authStore.isLogin) return false;
    if (typeof codes === 'string') {
      return authStore.userInfo.buttons.includes(codes);
    }
    return codes.some(code => authStore.userInfo.buttons.includes(code));
  }

  return { hasAuth };
}

// 组件中使用
const { hasAuth } = useAuth();
<ElButton v-if="hasAuth('system:user:add')">新增</ElButton>
```

|-----------------|--------|------|
| `token` | string | JWT 访问令牌 |

## 7. 多子系统权限隔离

### 7.1 应用层权限控制

用户可访问的子系统由 `sys_role_app` 关联链决定:

```
用户 → sys_user_role → 角色 → sys_role_app → 应用
```

**门户页面**只展示用户有权限的应用卡片，点击进入子系统。

### 7.2 子系统上下文管理

```typescript
// Auth Store
currentAppCode = ref('')  // 当前子系统编码

// 路由守卫 afterEach 自动检测
detectAppCodeFromRoute(path: string) {
  // /aps/*  → currentAppCode = 'APP_APS'
  // /mes/*  → currentAppCode = 'APP_MES'
  // /portal → currentAppCode = ''
}
```

### 7.3 侧边栏菜单按子系统过滤

路由的 `meta.appCode` 字段控制菜单在哪个子系统显示:

| 路由模块 | appCode | 说明 |
|----------|---------|------|
| 基础数据 `/basic-data` | `APP_APS` | 仅在 APS 系统侧边栏显示 |
| 交付策划 `/delivery-planning` | `APP_APS` | 仅在 APS 系统侧边栏显示 |
| 精准日计划 `/daily-plan` | `APP_APS` | 仅在 APS 系统侧边栏显示 |
| 产能规划 `/capacity-planning` | `APP_APS` | 仅在 APS 系统侧边栏显示 |
| 系统管理 `/manage` | *(无)* | 所有子系统共享 |
| 首页 `/home` | *(无)* | 所有子系统共享 |

**过滤函数**:
```typescript
function filterAuthRoutesByAppCode(routes, currentAppCode) {
  if (!currentAppCode) return routes;  // 门户不过滤
  return routes.filter(route => {
    const routeAppCode = route.meta?.appCode;
    return !routeAppCode || routeAppCode === currentAppCode;
  });
}
```

**响应式切换**: 通过 `watch(authStore.currentAppCode)` 监听，切换子系统时自动刷新侧边栏。

### 7.4 管理数据按子系统过滤

在子系统上下文中，管理页面的数据查询会带上 `appCode` 参数:

| 管理页面 | 过滤方式 | 说明 |
|----------|----------|------|
| 用户管理 | `GET /system/user/page?appCode=APP_APS` | 只显示该子系统角色关联的用户 |
| 角色管理 | `GET /system/role/page?appCode=APP_APS` | 只显示该子系统的角色 |
| 菜单管理 | `GET /system/menu/list?appCode=APP_APS` | 只显示该子系统角色关联的菜单 |

**后端过滤 SQL 示例** (菜单管理按 appCode 过滤):
```sql
SELECT DISTINCT m.*
FROM sys_menu m
INNER JOIN sys_role_menu rm ON m.id = rm.menu_id
INNER JOIN sys_role r ON rm.role_id = r.id
INNER JOIN sys_role_app ra ON r.id = ra.role_id
INNER JOIN sys_app a ON ra.app_id = a.id
WHERE a.app_code = #{appCode}
  AND m.status = '0' AND m.del_flag = '0'
  AND r.del_flag = '0' AND r.status = '0'
ORDER BY m.order_num
```

### 7.5 角色编辑跨系统保护

当在子系统中编辑用户角色时，只替换该子系统下的角色关联，保留其他子系统的角色:

```java
// SysUserServiceImpl.updateUser()
if (StrUtil.isNotBlank(dto.getAppCode())) {
    // 子系统模式：只删除该子系统下的角色关联
    userRoleMapper.deleteByUserIdAndAppCode(id, dto.getAppCode());
} else {
    // 全局模式：删除全部角色关联

```

```sql
-- 只删除指定子系统的角色关联
DELETE FROM sys_user_role
```

## 8. 多租户数据隔离

### 8.1 自动租户过滤

通过 MyBatis-Plus 的 `TenantLineHandler` 实现**自动 SQL 注入**:

```java
// MomTenantLineHandler
public Expression getTenantId() {
    Long tenantId = SecurityUtils.getTenantId();
    return new LongValue(tenantId != null ? tenantId : 0);
}
```

**效果**: 非管理员用户的所有 SQL 查询自动追加 `WHERE tenant_id = {当前租户ID}`。

## 9. 超级管理员保护策略

### 9.1 后端保护

#### 用户管理保护

```java
// SysUserServiceImpl
private void checkNotSuperAdmin(Long targetUserId) {
    if (!SecurityUtils.isAdmin()) {
        List<SysRole> targetRoles = roleMapper.selectRolesByUserId(targetUserId);
        boolean isSuperAdmin = targetRoles.stream()
                .anyMatch(r -> "admin".equals(r.getRoleKey()));
        if (isSuperAdmin) {
            throw new BusinessException("不允许操作超级管理员用户");
        }
    }
}
```

**保护范围**: `updateUser`、`deleteUser`、`changeStatus`、`resetPassword` 均调用此检查。

#### 角色管理保护

```java
// SysRoleServiceImpl
private void checkNotAdminRole(Long roleId) {
    if (!SecurityUtils.isAdmin()) {
        SysRole role = baseMapper.selectById(roleId);
        if (role != null && "admin".equals(role.getRoleKey())) {
            throw new BusinessException("不允许操作超级管理员角色");
        }
    }
}
```

**保护范围**: `updateRole`、`deleteRole`、`assignMenus` 均调用此检查。

### 9.2 前端保护

#### 用户管理页面

```tsx
// 禁用超级管理员的编辑/删除按钮
const isSuperAdmin = row.username === 'admin';
const currentIsAdmin = authStore.userInfo.roles.includes('admin');
const disabled = isSuperAdmin && !currentIsAdmin;

<ElButton disabled={disabled} onClick={() => edit(row.id)}>编辑</ElButton>
<ElButton disabled={disabled}>删除</ElButton>
```

#### 角色管理页面

```tsx
// 禁用超级管理员角色的编辑/删除按钮
const isSuperAdminRole = row.roleKey === 'admin';
const currentIsAdmin = authStore.userInfo.roles.includes('admin');
const disabled = isSuperAdminRole && !currentIsAdmin;
```

| 操作 | 非管理员 → 管理员 | 管理员 → 管理员 | 非管理员 → 普通用户 |
|------|:------------------:|:------------------:|:-------------------:|
| 编辑用户 | 禁止 | 允许 | 允许 |
| 删除用户 | 禁止 | 允许 | 允许 |
| 修改状态 | 禁止 | 允许 | 允许 |

## 10. 安全防护措施

### 10.1 请求头防伪造

网关在验证 JWT 后，**先清除**再**重新注入**内部请求头:

```java
// AuthFilter.java
// 1. 清除外部传入的内部头 (防伪造)
ServerHttpRequest.Builder mutate = request.mutate();
mutate.header(HEADER_USER_ID, "");      // 清空
mutate.header(HEADER_TENANT_ID, "");    // 清空

// 2. 从 JWT 中提取可信信息重新注入
mutate.header(HEADER_USER_ID, String.valueOf(userId));
mutate.header(HEADER_TENANT_ID, String.valueOf(tenantId));
```

### 10.2 密码安全

- **加密算法**: BCrypt (Spring Security PasswordEncoder)
- **盐值**: BCrypt 内置随机盐
- **验证**: `passwordEncoder.matches(rawPassword, encodedPassword)`

### 10.3 验证码保护

- **类型**: 4位字母数字混合
- **存储**: Redis (`captcha:{uuid}`)
- **有效期**: 120秒
- **一次性使用**: 验证后立即从 Redis 删除
- **可配置开关**: `mom.captcha.enabled`

### 10.4 跨域保护

```yaml
# Gateway 配置
mom:
  cors:
    allowed-origins: http://localhost:5173,http://localhost:9010,...
```

### 10.5 微服务间通信

Feign 请求拦截器自动传播用户上下文:

```java
// FeignRequestInterceptor.java
requestTemplate.header("X-User-Id", userId);
requestTemplate.header("X-User-Name", username);
requestTemplate.header("X-Tenant-Id", tenantId);
```

### 10.6 软删除机制

所有核心表使用 `del_flag` 字段实现软删除:
- `0` = 正常
- `1` = 已删除

MyBatis-Plus 全局配置逻辑删除，查询自动过滤已删除数据。
